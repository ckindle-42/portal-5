## TL;DR

oMLX v0.3.12 MTP speedup PASSES the 1.5× gate on all three output sizes
(1.55×-1.65×), but KV-cache warm TTFT is still WORSE than cold (same bug
class as the 2026-04-25 finding), concurrent throughput did not improve,
and the practical speedup over the production 4-bit baseline is marginal
(0.95-1.14×). Decision: **PROBE_AGAIN_NARROWLY** on MTP stability before
promotion; KV-cache finding warrants an oMLX issue.

## Required Judgment Questions

### Q1: Did warm TTFT beat cold this time? (The original cancel-trigger)

**No.** Warm TTFT is still WORSE than cold on oMLX, contradicting the
v0.3.8 release notes that claimed to fix the RotatingKVCache attention
dilution bug.

Per-model comparison:
- **Llama-3.2-3B**: cold 0.29s, warm avg 0.65s, delta **+124%** (warm slower)
- **Qwen3-Coder-30B-A3B**: cold 0.49s, warm avg 0.945s, delta **+93%** (warm slower)
- **Llama-3.3-70B**: cold FAILED (HTTP 507 memory), warm 22.27s/5.38s — unreliable data

For comparison, mlx-proxy shows the expected pattern (warm faster due to
model already loaded):
- Llama-3.2-3B: cold 0.28s, warm 0.175s, delta -37% (warm faster)
- Qwen3-Coder-30B: cold 0.74s, warm 0.61s, delta -18% (warm faster)
- Llama-3.3-70B: cold 4.40s, warm 1.075s, delta -76% (warm faster)

**Interpretation:** The KV-cache warm TTFT regression is the same bug class
as the 2026-04-25 finding. The v0.3.8 release notes described fixing
"attention dilution on RotatingKVCache SSD restore for sliding-window
models" but this is either (a) not the same bug, (b) not fixed for the
Qwen3.5 model type, or (c) a regression in v0.3.9-v0.3.12. The pattern
(warm 2× slower than cold) suggests the KV cache restore is injecting
stale or zero-padded positions into attention, exactly the bug class the
release notes describe. This warrants filing an oMLX issue with the exact
request shape used in this test.

### Q2: Did the 22 GB Qwen3-Coder-30B model load this time?

**Yes.** The v0.3.10 OMLX_MAX_PROCESS_MEMORY fix works. Qwen3-Coder-30B
(22 GB) loaded and produced TPS on oMLX.

- Model loaded successfully: yes
- TPS: 26.2 average across 9 runs (3 prompts × 3 runs)
- For comparison: mlx-proxy TPS = 25.0 on the same model

The 70B model (38.81 GB) is borderline — it loaded for TPS and concurrent
tests but failed the initial KV cache cold test with HTTP 507 (memory).
This confirms the memory accounting fix is working but the 70B model
pushes the M4 Pro 64 GB ceiling.

### Q3: Did MTP produce ≥1.5× TPS speedup at temp=0?

**Yes, on all three sizes.**

| Size | Control TPS | MTP TPS | Speedup ratio | ≥1.5× gate |
|---|---|---|---|---|
| short (128 tok) | 7.8 | 12.2 | 1.55× | PASS |
| medium (512 tok) | 8.2 | 12.9 | 1.57× | PASS |
| long (2048 tok) | 8.2 | 13.6 | 1.65× | PASS |

MTP acceptance rate from oMLX logs: 96.9% (31/32 draft tokens accepted
on the smoke test). The speedup increases with output length as expected
(MTP overhead is fixed per-request; benefit grows with output).

Comparison against the **production fleet 4-bit baseline** (different
quantization — captures the practical decision):

| Size | mlx-proxy 4-bit TPS | oMLX MTP TPS | Practical speedup |
|---|---|---|---|
| short (128 tok) | 11.8 | 12.2 | 1.04× |
| medium (512 tok) | 13.6 | 12.9 | 0.95× |
| long (2048 tok) | 11.8 | 13.6 | 1.14× |

**The practical speedup over the production baseline is 0.95-1.14×** —
effectively equivalent. The 8-bit MTP model vs 4-bit production model
is a significant confound: the 8-bit base model is inherently faster
(higher precision, simpler quantization math), and part of the "MTP
speedup" is really "8-bit vs 4-bit speed differential". The isolated MTP
speedup (control vs treatment on the same model) is real and clears the
gate, but the practical benefit is marginal.

**Quality contamination check:** At temp=0, the outputs for control and
treatment should be identical (lossless speculative decoding). The first
smoke test showed identical "Thinking Process:" prefixes for both
conditions, confirming MTP correctness. No quality drift detected.

### Q4: Did concurrent throughput improve on models that fit?

**No.** mlx-proxy remains faster under concurrent load on all models.

| Model | mlx-proxy Speedup | oMLX Speedup | Winner |
|---|---|---|---|
| Llama-3.2-3B | 2.70× | 2.44× | mlx-proxy |
| Qwen3-Coder-30B | 2.03× | 1.56× | mlx-proxy |
| Llama-3.3-70B | 2.04× | 1.85× | mlx-proxy |

The v0.3.9 PoolingCache / BatchPoolingCache changes did not translate to
concurrent throughput improvement on M4 Pro 64 GB. mlx-proxy's serial
execution is still faster, likely because oMLX's continuous batching
overhead exceeds the benefit on a single-user machine with no actual
concurrent traffic.

### Q5: Did oMLX crash, hang, or produce malformed outputs at any point?

**No crashes or hangs.** One HTTP 507 (memory) on the 70B model's cold
KV cache test — the model loaded for subsequent requests, suggesting the
507 was a transient memory pressure issue during initial load, not a
persistent failure.

- Crashes during run: 0
- Timeouts (REQUEST_TIMEOUT exceeded): 0
- Empty outputs: 0
- HTTP 507 errors: 1 (70B cold KV cache test)
- Server restarts required: 6 (3 MTP probe cycles × 2 restarts each,
  intentional for MTP toggle)

oMLX v0.3.12 is stable on M4 Pro 64 GB for models ≤30 GB. The 70B model
is borderline (occasional 507 under memory pressure).

### Selected cell: PROBE_AGAIN_NARROWLY

### Rationale

MTP speedup clears the 1.5× gate on all three output sizes (1.55×-1.65×),
and the v0.3.10 memory accounting fix successfully loads the 30B model that
OOM'd in the 2026-04-25 run. However, the KV-cache warm TTFT regression
persists (warm 2× slower than cold on both 3B and 30B), concurrent
throughput did not improve over mlx-proxy, and the practical speedup over
the production 4-bit baseline is marginal (0.95-1.14×). The MTP gate
passes in isolation, but the combination of broken KV cache and marginal
practical benefit does not warrant full promotion. A narrow stability probe
(100+ sequential MTP requests) is the right next step: if MTP is stable
under sustained load, it can be promoted as a side-car for long-output
coding tasks where the 1.65× speedup at 2048 tokens is meaningful.

## Limitations

- Single measurement per cell in MTP phase. Variance unknown. Production
  promotion would require N≥3 with median.
- 4-bit vs 8-bit-MTP base model is not an apples-to-apples MTP isolation.
  The 8-bit non-MTP MLX equivalent of Qwen3.6-27B (if published) would
  tighten the test. The `Jundot/Qwen3.6-27B-oQ4` exists but does NOT
  carry MTP weights — only `-mtp`-suffixed Jundot variants do.
- macOS 26.5 (Tahoe) required for oMLX. Tested on macOS 26.5 only.
- M4 Pro 64 GB only. M3 Ultra 512 GB results published by the oMLX
  maintainer are not transferable. The Ultra has 800 GB/s memory
  bandwidth vs M4 Pro's 273 GB/s — a 3× gap that disproportionately
  affects MoE decode and large-prompt prefill.
- One run, one day, one hardware unit. If the result is borderline,
  schedule a follow-on run at a different time of day to account for
  thermal variation.
- The bakeoff and MTP phases were run separately (bakeoff first, then
  MTP probe with oMLX restarts). Memory state may differ between phases.
- MTP toggle required oMLX restart (~25s each). This is a per-model
  setting, not a per-request parameter, which limits the granularity of
  A/B testing.

## Decision

**PROBE_AGAIN_NARROWLY.** MTP speedup is real and clears the gate, but
the KV-cache regression and marginal practical benefit over the production
baseline do not warrant immediate promotion. Next step: write
TASK_OMLX_MTP_STABILITY_V1.md (100+ sequential MTP requests at long
output size, measuring crash rate and speedup degradation). If stable,
promote oMLX as an MTP side-car for long-output coding tasks.

Separately, file an oMLX issue for the KV-cache warm TTFT regression on
Qwen3.5 model type, with the exact request shape from this test as
reproduction steps.
