## UAT Findings ‚Äî Issues Requiring Action

### FINDING-01: AEON Model Routes to Code Execution Instead of SearXNG for Web Search (T-12)

**Severity**: HIGH ‚Äî tool routing defect  
**Test**: T-12 Web Search ‚Äî Recent CVEs via SearXNG  
**Workspace**: auto-security | **Model**: Qwen3.6-27B-AEON-Ultimate-Uncensored-BF16-mlx-4Bit  

**What happened**: The model was asked to search for recent CVEs using SearXNG. Instead of invoking the SearXNG web search tool, it attempted to use `portal_code_execute_python` and `portal_code_execute_nodejs` to fetch CVE data from `https://api.circl.lu/v1/cves/` via generated scripts. Both scripts contained syntax errors:

- Python: `SyntaxError` in output formatting  
- Node.js: `await` used outside async function  

The model then fell back to its training knowledge to provide CVE data (CVE-2024-53183, CVE-2024-3116, CVE-2024-2024). The test assertions passed (CVE IDs present, len=11045) because the model used its own knowledge, but the **SearXNG search pipeline was never invoked**.

**Impact**: The auto-security web search capability is non-functional when AEON is the routing model. The model generates broken code execution scripts instead of using the available SearXNG tool.

**Evidence**: Chat `http://localhost:8080/c/6972bac5-9c3e-46c7-9025-7d2f6dea999e` ‚Äî assistant response explicitly states "the attempts to retrieve the CVE data failed due to syntax errors in the execution scripts [2][3]". Pipeline logs confirm `portal_code_execute_python` and `portal_code_execute_nodejs` were invoked, not SearXNG.

**Recommended action**: Investigate auto-security workspace tool binding ‚Äî AEON should be directed to use SearXNG (`portal_searxng_search` or equivalent) for web queries rather than generating ad-hoc code execution scripts. Consider adding an explicit tool preference or system prompt guidance.

### FINDING-04: Thinking Model Premature DOM Stability (Systemic)

**Severity**: MEDIUM ‚Äî driver timing  
**Affected**: All tests using Qwen3.5-9B-abliterated or AEON 27B  

These models emit "Thinking Process:" / `<details type="reasoning">` blocks that make the Playwright DOM appear stable while the model is still generating. The driver detects "stream complete (DOM stable)" in 3-6s and exits, then finds no content via API. Recovery happens on retries 2-3 when the model has finished. Tests using these models consistently show 1-2 "empty response" retries before passing.

### FINDING-05: ComfyUI Keeps Restarting Across Phases

**Severity**: LOW ‚Äî operational noise  

ComfyUI was killed by the inter-phase gate at every gate transition (Gates 1-6) because it runs host-native and the gate treats it as a non-MLX memory consumer. Each kill required a 30s Metal buffer release wait. The `--keep-comfyui` gate flag for Phase 5‚Üí6 was used correctly, but ComfyUI is restarted by launchd/monitoring between gates. Consider making ComfyUI persistent across non-media phases if it doesn't consume significant GPU memory when idle.
